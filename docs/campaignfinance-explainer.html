Overview

One schema for many different filings

Every state publishes campaign-finance data in its own shape. This system collapses all of it into a single unified_* schema, then runs entity resolution to turn millions of raw contributor and committee records into a deduplicated canonical reference. Data moves through four layers — the first three are mechanical; the fourth is where the hard problem lives: deciding that "Bob Smith" and "Robert Smith" at the same address are one person.

The upload & deduplication process

Dedup happens twice — on purpose

The cheap pass runs as data lands: an in-memory BuilderCache does get-or-create on natural keys, writes go through a dialect-aware bulk_upsert, and partial unique indexes backstop it. The expensive pass runs afterward: probabilistic entity resolution that catches the fuzzy duplicates exact keys never will.

Entity resolution · app/resolve

The seven-stage match run

One ResolutionRun orchestrates the lifecycle — running → completed, or running → failed. A failed run rolls back any partial canonical writes under a delete-and-replace contract, so no half-resolved state survives.

1

Standardize

→ resolution_input

Normalize names, addresses, and org strings into comparable form; build one input row per source record.

2

Blocking

→ candidate_pairs

Generate only the pairs worth comparing using blocking keys — never the full N² cross product.

3

Fast-path

→ merge_edges · match_decision

Deterministic exact-and-rule matches short-circuit straight to a decision, skipping the scorer.

4

Splink score

→ scored_pair

Probabilistic record linkage scores each remaining candidate pair with an explainable match weight.

5

Classify

→ match_decision · merge_review

Score bands split pairs three ways: auto-merge, queue for human review, or reject.

6

Cluster

→ clusters

Connected-components clustering groups all transitively-matched records into one entity, with a max-cluster-size guard.

7

Survivorship + publish

→ canonical_entity · crosswalks · views

Pick surviving field values per cluster, write the canonical master and crosswalk, then refresh the publish views.

Load-time dedup · the cheap pass

Keys that collapse exact duplicates

BuilderCache — in memory

One read-through cache per load run turns repeated person, committee, entity, and address lookups into dict hits instead of per-row SELECTs. Each kind has a natural key.

kinddedup key
person (individual)lower(first), lower(last), state, addr_key
person (org)lower(org), state
entitytype, normalized_name, state
addressstreet, city, state, zip (≥2 set)
campaignname, committee, candidate, year

Partial unique indexes — in the database

The keys are enforced by raw, idempotent DDL so the cache and a concurrent writer can never disagree. The upsert resolves conflicts on these same columns.

indexguards
uix_persons_name_stateindividuals by name + address

Why two passes

Exact keys are fast and safe but blind to spelling. Splink is expensive but catches "Bob" vs "Robert," abbreviations, and typos. Running the cheap pass first means the resolver only reasons about the genuinely ambiguous records.
